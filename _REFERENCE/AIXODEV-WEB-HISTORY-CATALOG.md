## 0. Orientation — What `aixodev-web` Is, and Its Phase Skeleton

`aixodev-web` is the **server-side web application** of the AIXO.Dev Platform: a Flask + PostgreSQL + HTMX/Tailwind dev-project-management platform (issues, wiki, GitHub, analytics, notifications) that also ingests and renders AI coding-session transcripts (Claude Code JSONL, Claude.ai exports), and receives real-time hook events from the companion `aixocode` CLI client. Tech stack as built: Python 3.11 / Flask 3.x / SQLAlchemy 2.0 / PostgreSQL 16 / Redis 7 / Flask-SocketIO (eventlet→gevent→threading over time) / HTMX 2.0 / Tailwind 3.4 / Chart.js 4 / highlight.js / Dramatiq+Periodiq workers / `uv`+`pyproject.toml`+`pydantic-settings`+Ruff/mypy/pytest. ## PHASE 5 — Entity Model & Platform Vision (4 sprints shipped, IN PROGRESS)

> A research-heavy phase that **specified** (did not implement) an expansion from 25→~75 tables across 12 entity families, and **built** supporting infrastructure (debugging, ops, staging, session-ingest API) operationalizing the `aixodev-web` (server) ↔ `aixocode` (CLI client) split. (Source: `phase_05--entity_model_and_platform_vision/`)

**`DECISIONS.md` is effectively empty** (placeholder only); the real architectural decisions live in §16 of the entity-model spec:
- **Hierarchy strategy = ltree** (PostgreSQL `ltree` for deep/queried hierarchies — projects/topics/dev_tasks/wiki; adjacency list for shallow ones; both representations coexist).
- **Agent vs User = SEPARATE entity** (agents have model/persona/routing, token budgets, distinct audit/billing/retire; unified in UI via `DevTeamMembership.member_type='person'|'agent'`).

- **Polymorphic pattern** = string `entity_type` discriminator + nullable per-type FK.
- **Naming conventions** = plural snake_case tables, singular PascalCase models, VARCHAR(20) status strings (not enums).
- **Client-server separation** (recorded in `sp_04` + `_research/ARCHITECTURE_ANALYSIS--aixocode-vs-aixodev-web.md`): `aixodev-web` = authoritative Flask/PG server; `aixocode` = per-developer CLI doing hook capture + JSONL parse + SQLite archival + queued sync.

Columns: `id` Integer PK · `project_id` Integer FK `projects.id` `ON DELETE CASCADE` NOT NULL indexed · `url` String(500) NOT NULL (service requires http/https scheme) · `name` String(200) NOT NULL · `default_branch` String(100) NOT NULL default `'main'` · `github_owner` String(100) nullable · `github_repo` String(100) nullable · `created_at` DateTime NOT NULL. Normalized association table (ADR-001 chose this over a Postgres ARRAY).

**Hierarchy mechanism (the key question):** a single self-referential **adjacency-list FK** `projects.parent_id → projects.id` `ON DELETE SET NULL` with a `children` backref. - **Why graft `aixodev-workgroups`' Project/Repository schema on top:** Its design is the most sophisticated of the three and is *more correct for GEN2* than what `aixodev-projects` ships — it splits the **repo entity (`repositories`, shareable across projects, with `CHECK(local_dir OR remote_url)`)** from the **project↔repo association object (`project_repositories`, carrying `role` + `is_primary`)**, and adds the requirement-driven **`projects.local_path`** (durable working dir set before any git repo exists). `aixodev-projects`' `ProjectRepository` is a simpler 1-to-N (a repo belongs to exactly one project, no sharing). If GEN2 needs repo-sharing/monorepo modeling, lift the workgroups three-table design (copy-pasteable ORM in `DEV_PLAN.md` §1.1) and replace the `aixodev-projects` `ProjectRepository`; if not, keep the simpler shipped one and just add `local_path`. Either way it is a *schema design to implement*, not running code.
- **Why NOT `aixodev-codemap` as the base:** Its app skeleton is excellent (arguably the single cleanest CRUD blueprint + config trio of the three), but its domain model is a code-scanner (`CodeMapProject` ≈ a scanned codebase, no real Repository entity, scan-version-scoped not org-hierarchy). You'd keep the plumbing and discard all domain tables — and `aixodev-projects` already gives you that *same* plumbing PLUS a real Project/Repository/techstack slice. Borrow individual patterns from codemap (the `{"data","message","errors":[]}` API envelope, the SAVEPOINT test fixture, the `ScanVersion` denormalized-pointer versioning idiom if GEN2 wants snapshots), but don't base on it.

**What GEN2 must add net-new on top of the `aixodev-projects` base (none of these exist as data in ANY of the three repos):**

- **Build-Line · Build-Envelope · Stage · Phase · Sprint · Milestone** — the entire Project-Organization-Model hierarchy (Project → Build-Line → Build-Envelope → Stage → Phase → Sprint → Milestone). Absent everywhere as data (they appear only as docs/process/git-conventions). New tables required; the existing `Project.parent_id` adjacency-list-with-service-layer-cycle-guard is the proven local pattern to follow for each level, or model the chain as explicit FK-linked tables.
- **Idea · Topic** — absent everywhere (not even in spec). Fully net-new (note: the separate `kingstrat`/Divia ideation work models these elsewhere — check before designing from scratch).
- **techstack as a first-class entity** — only a precursor exists (`aixodev-projects.ProjectLanguage` name/kind/is_primary). Promote to a real `TechStack` table if needed; the `ProjectLanguage` shape is the starting point.
- **Repository entity upgrade** — graft the `aixodev-workgroups` `repositories` + `project_repositories` association-object design (+ `projects.local_path`) per above.
- **Auth** — none of the three ship auth (all carry a nullable `created_by`/no-auth placeholder); add it fresh when GEN2 needs it.

**One-line summary:** lift `aixodev-projects` as the running base (Project + ProjectRepository + ProjectLanguage CRUD + the whole Flask/SQLite scaffold + test harness), strip its theme subsystem, graft the `aixodev-workgroups` repo/association schema design on top, and design the Build-Line → Milestone org-hierarchy plus Idea/Topic/first-class-techstack entities net-new — borrowing codemap's API-envelope and test-fixture patterns where convenient.
