# Idea — AI contacts / relationship manager (CRM-in-the-inbox)

> Layer-A Idea node (durable · brand-free). Model: [`STRATEGIC-LANDSCAPE-MODEL.md`](../../STRATEGIC-LANDSCAPE-MODEL.md). **Bootstrap seed.**

- **Topic(s):** [contacts-and-crm](../TOPICS/contacts-and-crm.md) (+ [knowledge-graph-entity-modeling](../TOPICS/knowledge-graph-entity-modeling.md), [ai-agents-orchestration](../TOPICS/ai-agents-orchestration.md))
- **One-line:** A thin CRM-style reader that, where people already work (starting in Gmail), resolves senders to their People/Companies entities in a PKMS, surfaces and manages that entity's tasks/events, logs activity back, and runs a relationship-cadence review ("you haven't replied to X in three weeks").
- **Axes (Idea-level):** Conviction = **active** (Gmail research complete — 16 tracks, ~69k words — but pre-Phase-00, no code) · Horizon = **now / v1** for the Gmail reader; **later** for mobile + the relationship-cadence agent · Provenance = John's portfolio vision; explicit model is **Streak/Copper** (CRM-in-Gmail) · Leverage = the **"thin renderer, server-side intelligence"** pattern (new agent capabilities ship as server deploys, no Chrome-review latency); the org task graph as the moat; the swappable `DiviaBackend` (Stub ↔ Http).
- **Idea↔Idea edges:** `depends-on` the PKMS / graph-DB entity engine (it is a thin renderer; the heavy PKMS stays in Professional + Enterprise) · `enables` Capture-to-Activity-Log → the Divia.Network fan-out (an email becomes a DiviaCard) · `sibling-of` [AR storefront-discovery](ar-storefront-discovery.md) (DiviaContacts' job extends from inbox People/Companies to physical-world Places) · relates-to [personal life-OS](personal-life-os-gtd.md) (the consumer-side relationship-cadence twin)
- **Channel (Idea→Venture, time-bounded):** [DiviaContacts](../PRODUCTS/DiviaAI/diviacontacts.md) (🟠 Gmail research complete, pre-code; `diviacontacts-gmail` real, `-android`/`-iOS` empty placeholders). Target market post-pivot: **Divia.AI Enterprise SMEs**.
- **Realized by (Build Lines, Layer B):** TBD — a Chrome MV3 + InboxSDK extension (Preact + TS, Shadow DOM) is the decided architecture for the Gmail edition; no Build Lines defined.
- **Notes:** Identity spine: per-mailbox `(account_email, gmail_thread_id)` + RFC 5322 `Message-ID`. Link-don't-copy privacy (IDs + metadata + optional snippet, never bodies). Frontier ideas: cross-mailbox team awareness, a closed `waiting`-on-reply loop, PARC TaskMaster "thrasks," an agent-accessible task graph via MCP.
